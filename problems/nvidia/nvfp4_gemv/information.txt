🔧 1. What mma.sync.aligned.m16n8k16.row.col.f16 Actually Does

This instruction performs:

A matrix multiply-accumulate:
[16x8] = [16x16] × [16x8],
using FP4 → FP16 converted values, and accumulates into FP16 output.

Operand A: [16 × 16] FP4 → FP16 (row-major)

Operand B: [16 × 8] FP4 → FP16 (col-major)

Output: [16 × 8] FP16 tile, distributed over a warp (32 threads)

Each thread contributes partial sums (accumulation) over the K dimension of 16.

🧩 2. Fragment Layout for m16n8k16

Each warp (32 threads) computes a full [16x8] output tile. Threads are grouped like so:

Rows: 16 — partitioned into 8 quads (0 to 7), each covering 2 rows.

Cols: 8 — each lane handles 2 columns.

So:

Each thread covers 2 output elements: one for row r0, and one for r1 = r0 + 8, across the same output column.

Each fragment like c_frag_0, c_frag_1, ... holds two FP16 values for that thread.

From mma.sync result:

fragment<accumulator, 16, 8, 16, half> c_frag;


Each thread’s registers:

c_frag.x → output element at [r0][c]

c_frag.y → output element at [r1][c] (i.e. row+8)

For mma.sync, we typically use 4 fragments: c_frag_0 ... c_frag_3
These map to each of the 4 columns in [16x8] tile that the thread processes 2 rows for.

🧵 3. How Each Thread Contributes to c_frag_0...3

Let’s break it down:

fragment<accumulator, 16, 8, 16, half> c_frag_0, c_frag_1, c_frag_2, c_frag_3;


These correspond to the 4 column-pairs (2 per thread) in the 8-column tile.

Each thread covers 2 output rows and contributes to 4 elements (2 rows × 2 cols):

Fragment	Row (quad)	Col offset
c_frag_0	r0 / r1	col 0
c_frag_1	r0 / r1	col 1
c_frag_2	r0 / r1	col 2
c_frag_3	r0 / r1	col 3

So a lane owns a 2×4 submatrix of the 16×8 tile.
