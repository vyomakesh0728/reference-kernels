2. Where the Error Appears
Output mismatches such as ERROR AT 0, 0, 0 0.0 4668.0 (and high numbers) are observed at the final output buffer, meaning the computed values written by the kernel do NOT match the reference even though decode and scaling are correct.
Large errors—like 0 vs 4000+—point to either:
The wrong values are being written for some output rows/columns, or
The MMA tiles are not being mapped to the output buffer the same way as reference.


3. What Causes This
Tensor core (mma.sync, ldmatrix, scatters) path is highly sensitive to lane arrangement and warp output mapping.
The way the MMA fragments are loaded, accumulated, and then written back ("scatter") must exactly match NVIDIA's/CuTe/CUTLASS expected output layout.
If there is any mismatch in the:
loading logic for A and B (ldmatrix, .aligned.m8n8, etc.)
warp lane to output row/column mapping (the “scatter”),
or if output buffer indices are calculated incorrectly,
…then huge differences arise, even if the dot products and per-FP4 math are otherwise correct.


4. Supporting Clues
Your kernel does an in-kernel debug of a naive sum (i.e., a simple K loop with the decoded FP4 and scale values) alongside the MMA result.
When you compare [KERNEL NAIVE] to the reference, the naive result matches the reference, but the tensor core result diverges.
This is rock-solid confirmation: decode+scale are totally correct. The MMA/scatter is the culprit.


5. What to Do Next
Don’t change scale decode/indexing again—it’s already fixed.
Focus on:
Mapping how the tensor core accumulators are written to C/output: does your logic for “which lane writes which element” match the reference?
Does each warp always write its output to the correct global row in the matrix C?
Is the mma.sync, ldmatrix load (row/col order), and scatter block mapping correct, especially for edge cases and padded regions?
As a debug aid, try running your kernel with the tensor core code temporarily replaced by the naive sum for all outputs. If that matches, it's purely a tiling/scatter bug.
