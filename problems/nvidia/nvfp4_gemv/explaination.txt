Output Writeback Bug in submission.py (Tensor Core Kernel)

The root cause of the mismatches is a faulty output scatter (writeback) logic in the tensor-core kernel of submission.py. In particular, the mapping from each warp’s partial results to the global output matrix is incorrect, causing many output rows to be never written or written with wrong values (zeros). The bug lies in how the code assigns two output rows per thread (in a 2×2 fragment) – it assumes the two rows are separated by 8, which is misaligned with the actual tensor core output layout.

Symptom: Many Outputs Left as Zeros (Even-Indexed Rows Missing)

In tests (e.g. M=7168, L=1, K=16384), submission.py produced widespread mismatches against the reference. Almost every even-indexed output row was zero while the reference had large nonzero values, whereas odd-indexed rows had nearly correct values (with minor differences). For example, row 0 came out 0.0 vs reference 4668.0, row 2 was 0.0 vs 4756.0, row 4 0.0 vs 4812.0, etc., while row 1 and 3 had nonzero values close to expected【10†】. This pattern indicates that entire sets of rows were never properly written, rather than a random arithmetic error. The decode/scale logic was correct (confirmed by debug for first row), so the fault had to be in the final accumulation or indexing of results per tile.

The Faulty Writeback Logic

Inside the CUDA kernel fp4_gemv_sm100_ptx_mma, after performing all MMA operations for a tile, the code scatters each thread’s computed 2×2 output fragment to global memory. It divides the warp’s 32 threads into 8 groups (“quads”) of 4 threads (quad = lane_id >> 2), and each such group is intended to correspond to a pair of output rows. Only one thread per group (the one with col_in_quad == 0, i.e. lane indices 0,4,8,12,… within each group) writes results for two rows (using its accumulator registers c_frag_0 and c_frag_2). The critical bug is in how it computes those two row indices:

int quad = lane_id >> 2;       // 0..7, identifies which pair of rows
int col_in_quad = lane_id & 3; // 0..3, identifies which pair of columns
if (col_in_quad == 0) {
    int r0 = quad;
    int global_row0 = m_cta + row_offset + r0;
    int global_row1 = global_row0 + 8;
    if (global_row0 < M) D_batch[global_row0] = __float2half(c_frag_0);
    if (global_row1 < M) D_batch[global_row1] = __float2half(c_frag_2);
}


Here row_offset = warp_id * 16 and m_cta = blockIdx.x * 64 (tile starting row). The code assumes that for each “quad” of 4 threads, the two output rows are r0 = quad (within the warp’s 16-row tile) and r0 + 8. This is wrong. In a 16×8 output fragment produced by one warp, a given thread’s 2×2 fragment does not span rows that are 8 apart in the final result. The warp’s 16 output rows are actually laid out differently (contiguously in pairs). In other words, the mapping global_row1 = global_row0 + 8 is a misinterpretation of the tensor core MMA output indexing.

Why This Causes Missing/Zero Outputs

Because of the above misalignment, half of the output rows in each tile never receive the correct data:

Even-indexed rows (e.g. 0,2,4,…) came out zero – these correspond to output positions that the code never wrote the proper result to. For instance, in warp0, the code wrote results for row0 and row8 using thread0. But if the actual second row of thread0’s fragment was row1 (not row8), then row0’s partner result went to the wrong index (row8) and row1’s result was never stored in row1. More generally, each thread wrote to row + 8 which often was not the actual intended second row of its fragment. This left many true output rows unwritten (remaining at 0), particularly the even-indexed ones in the observed pattern.

Odd-indexed rows had values (slightly off) – these got some data written (e.g. thread4 wrote to row1, thread12 to row3, etc.), but those values were misassigned. In our example, thread4 assumed it was writing the first column result of row1 (which happened to be nearly correct), and row1 did get a value ~2–3% off from reference. However, the slight differences on odd rows are likely due to numerical issues (or partial accumulation differences) – the big red flag is the zeros on the other half of the rows.

In short, the output tiling logic effectively skipped every other row (writing them to the wrong index or not at all). This aligns with the 7118 mismatches out of 7168 for M=7168: almost all even rows (3584 of them) were completely wrong (zero), and the odd rows were off enough to count as mismatches too【10†】.

Comparison with Correct Implementation (submission2.py)

The reference/naive implementation does a straightforward write of each output element, and the second optimized implementation (submission2.py with full FMA) presumably uses a correct mapping. By comparing, one finds that submission2.py writes each output row exactly once in order, whereas submission.py tries to pack two outputs per thread with an incorrect offset. There is no issue in the decoding, scaling, or shared memory loading – those steps match reference. The entire discrepancy comes from this final reduction/indexing bug.

Conclusion: The Exact Bug and Fix

The bug is the misaligned row index calculation in the tensor-core kernel’s output scatter phase. The code assumes each warp thread’s 2 results are row = r0 and row+8 within the tile, which is wrong. The problematic lines are the ones computing global_row1 = global_row0 + 8 (in the if (col_in_quad == 0) block). Because of this, not all tiles’ rows are written to D correctly – many remain zero or are placed in the wrong slot.

To fix it, the output mapping logic must be corrected to the actual warp fragment layout. For example, if each “quad” of 4 threads actually corresponds to two consecutive output rows, then r0 should be used differently (likely r0 = quad * 2 and global_row1 = global_row0 + 1). In summary, the second row index calculation is wrong – adjusting that to the true spacing (instead of 8) will ensure every output row gets its accumulated value. This one-indexing bug in the tensor core MMA reduction is the cause of the widespread mismatches in submission.py.
